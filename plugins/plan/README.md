# plan

A lightweight, markdown-based bug/issue/chore/todo tracker that lives inside your repo at `docs/plan/`. One file per item, YAML frontmatter for structure, free-form markdown body for everything else. No external service, no database — just files git tracks.

## Commands

| Command | What it does |
|---|---|
| `/plan-init` | Create the `docs/plan/` folder and seed `README.md`. Optional — `/plan-add` auto-inits. |
| `/plan-add <title>` | Create a new item. Inline modifiers: `type:bug`, `priority:high`, `due:friday`, `tag:auth`. |
| `/plan-list [filters]` | List items, filtered. Defaults to non-done. Filters: `status:`, `type:`, `priority:`, `due:`, `tag:`. |
| `/plan-update <id-or-slug>` | Change a field or append a dated note. Modifiers same as `/plan-add` plus `status:`, `title:"new"`, `untag:`, and `note:"text"`. |
| `/plan-close <id-or-slug>` | Mark done and stamp the closed date. Optional `reason:"text"`. |

## Item shape

Each item is `docs/plan/NNN-kebab-slug.md`:

```markdown
---
id: 042
title: Add search to docs site
type: feature           # bug | feature | chore | todo
status: open            # open | in-progress | blocked | done
priority: med           # high | med | low
due: 2026-06-01         # ISO date, optional
tags: [docs, search]    # optional
created: 2026-05-18
updated: 2026-05-18
closed:                 # set on close
---

A short description of what needs to happen and why.
```

`docs/plan/README.md` is auto-regenerated by every `/plan-*` command and renders as a clickable index on GitHub.

## Why this shape

- **Flat folder, single file per item** — closing an item flips frontmatter, never moves the file. `git log <file>` stays meaningful, PR links don't rot, `grep` is trivial.
- **Sequential 3-digit IDs** — short, sortable, typeable. `/plan-close 7` works.
- **Minimal frontmatter** — four statuses, four types, three priorities. Enough to plan, not enough to bikeshed.
- **README as the dashboard** — full rewrite each command, no patch logic, merge conflicts just resolve by re-running any `/plan-*`.

## Installation

```bash
/plugin install plan@mikersays-plugins
```

## Usage example

```
/plan-init
/plan-add Fix login redirect on Safari type:bug priority:high due:friday
/plan-add Add dark mode toggle type:feature tag:ui
/plan-list status:open type:bug
/plan-update 1 status:in-progress note:"reproduced; cookie domain mismatch"
/plan-close 1 reason:"shipped in v2.4.1"
```

## Constraints

- Never commits or pushes. The user reviews diffs and ships when ready.
- Never deletes or moves item files. Closed items stay forever as history.
- Refuses to operate outside a git repo — planning history without version control is worthless.
- macOS and Linux supported (BSD `date` and GNU `date` both handled). Windows untested.

## Pairing with the rest of the marketplace

After triaging with `/plan-*`, ship the diff with `/ship` and open a PR with `/pr`. If you want to mirror items to GitHub Issues, `gh issue create --body-file docs/plan/NNN-*.md` does the job — that's a one-off command, not built into the plugin.
